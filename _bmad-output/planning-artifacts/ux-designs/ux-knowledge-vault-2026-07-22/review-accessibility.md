# Accessibility Review — Knowledge Vault

## Overall verdict

The diagnosis is sound and the spine (mutual-exclusive panes, viewport-capped textarea, always-reachable row actions, tap fallback for move) correctly eliminates the two reported bugs. But the spec's accessibility floor is declared once in DESIGN.md and then under-operationalized where an implementer will actually read instructions: the 44px touch-target commitment is never restated with a concrete Tailwind treatment against the three controls it names, one row-action trigger has zero accessible name today (confirmed by reading the code, not hypothesized), and the `(hover: hover)` gate has a real hybrid-device gap that can silently resurrect the exact bug this audit exists to fix.

## Findings

- **high** `FolderMenu`'s trigger button (`Explorer.tsx:654`) has no accessible name at all — no `aria-label`, no `title`, no visually-hidden text, just a bare `<MoreHorizontal>` icon inside `<button className="opacity-0 group-hover:opacity-100">`. This is a confirmed WCAG 4.1.2 (Name, Role, Value) failure today, not a maybe. EXPERIENCE.md's Accessibility Floor bullet 3 hedges ("gain a real `aria-label`… verify at implementation time, not assumed here") even though the audit already had this exact line open (it's quoted verbatim in Component Patterns for the opacity finding). *Fix:* state definitively in the spec that this button needs `aria-label="Mais ações"` (or per-item "Ações de {itemName}") — don't defer a fact you already verified.

- **high** The 44px `touch-target-min` token is declared in DESIGN.md and explicitly named against "the pane toggle, row-action buttons" (Layout & Spacing) and against `pane-toggle.minTouchTarget` in frontmatter — but EXPERIENCE.md's Component Patterns Fix column for Pane Toggle, the Editor header toolbar (save button), and Row actions never states a size/padding change. All three stay implicitly at their current shadcn sizes: `PaneToggle`/save button use `size="sm"` (`h-8`/32px, confirmed in `button.tsx`), and the row-action trigger has no size classes at all (hit area ≈ the 14px icon). DESIGN.md even promises "EXPERIENCE.md's Component Patterns section flags which ones need a mobile-only bump" — it doesn't. *Fix:* add an explicit line to each of these three Fix-column entries (e.g. "pad to `min-h-11 min-w-11` below `md:`, glyph stays visual-only size") so the 44px floor survives contact with the table an implementer will actually follow.

- **high** State Patterns' `(hover: hover)` / `(pointer: coarse)` split doesn't account for hybrid-pointer devices (Surface-type 2-in-1s, many Chromebooks) that have both a touchscreen and a trackpad. Such devices commonly report `hover: hover` because *some* attached input supports hover, even when the user is actively tapping the touchscreen. At `≥ md` with `(hover: hover)` the spec says row actions revert to "Unchanged: hover-to-reveal, as today" — i.e. the exact hover-only bug this audit exists to fix comes back, silently, on real hardware, at a viewport width the spec otherwise treats as "fixed." The `focus-within` fallback (Accessibility Floor bullet 4) doesn't rescue this case either — it only fires when the trigger itself receives focus, and a touch tap on an `opacity-0` element doesn't reliably do that. *Fix:* note the residual risk explicitly and consider `any-pointer: coarse` (or keeping the always-tappable treatment whenever *any* touch input is present, not just when the primary pointer is coarse) rather than `pointer: coarse` alone.

- **medium** No `aria-live` (or equivalent) announces the pane swap when the single visible pane changes below `md:`. `aria-pressed` on the toggle button is correct and sufficient for a screen-reader user whose focus is still on that button, but if they've moved focus into the pane content (e.g. tabbed to a wikilink inside Preview) before toggling, the entire content region silently replaces under them with no notification — disorienting given how structurally different raw Markdown vs. rendered HTML headings/lists are. *Fix:* a small `aria-live="polite"` status (e.g. "Mostrando: Preview") near the toggle, or `role="status"` on the pane container.

- **medium** Focus can be dropped to `<body>` when a pane holding the currently-focused element is unmounted. Because `showRaw`/`showPreview` conditionally mount/unmount each pane's DOM subtree, if a keyboard/AT user has focus on something inside Preview (a wikilink `<Link>`) and the *other* toggle is activated (by mouse, or by another input path), React unmounts that subtree and the browser has nothing to move focus to — it falls back to `<body>`, losing the user's place with no signal. EXPERIENCE.md's State Patterns note "no scroll-position carry-over between panes" but say nothing about focus. *Fix:* on toggle, if `document.activeElement` was inside the pane being hidden, move focus to the newly-shown pane's toggle button (or its first heading) instead of letting it fall through.

- **medium** The "Mover para…" folder-picker's keyboard path is unspecified. Key Flow 2 and the Interaction Primitives section describe only the tap sequence (tap kebab → tap "Renomear"/"Mover para…" → picker opens). Nothing states whether the picker is a focus-trapped dialog, where initial focus lands, whether Escape closes it, or whether focus returns to the triggering row's button on close — all of which matter for a keyboard-only or switch-access user performing the same "reorganize files" flow Ana does by touch in Flow 2. *Fix:* one line in Accessibility Floor or Interaction Primitives committing to standard dialog focus-management semantics (trap, Escape-to-close, return-focus-to-trigger) for the new picker.

- **low-medium** Component Patterns' "Explorer / FolderTree row actions" table row conflates two different components under one Fix. `FolderTree.tsx` is explicitly a read-only curation tree ("Deliberately separate from Explorer's editable tree — no rename, drag, or create," per its own top-of-file comment), and its only real caller (`admin.tsx` / `workspace-index.tsx`) wires `docAction` to a **Publish** button, not rename/delete/move. Only `Explorer.tsx:654`'s `FolderMenu` actually has rename/delete/move. The spec's shared Fix description ("Row actions become always-visible… persistent, tappable kebab") and the Accessibility Floor's aria-label instruction risk being applied to the wrong trigger, or an implementer building a "Mover para…" entry into the admin-only read-only tree by mistake. *Fix:* split the table row — `FolderTree.tsx:143`'s existing `focus-within` fallback is already correct and needs no behavior change (only its Publish-button's own label to verify); `Explorer.tsx:654` is where the real rename/delete/move/opacity/aria-label work belongs.

- **low** Raw Markdown `<textarea>` (`Editor.tsx`) has no accessible label — it relies on `placeholder` alone ("Escreva em Markdown…"), which most AT treats as a weak fallback name, not a real label. Pre-existing, not introduced by this pass, but the spec is already touching this exact element (for the height-cap fix) — cheap to bundle `aria-label="Markdown bruto"` into the same edit. *Fix:* add the label alongside the height-cap change since the element is already being modified.

- **low** Motion: the only transition implied is the row-action opacity fade (already `transition-opacity` in `FolderTree.tsx` today, presumably carried into the `Explorer.tsx` fix). This is subtle enough (opacity only, no transform/motion) that it doesn't warrant a `prefers-reduced-motion` note at this scope — confirmed non-issue, not a gap worth spec text.

## What's already solid

- The diagnosis is evidence-grounded, not speculative — every Component Patterns row cites the actual offending code (`showRaw`/`showPreview` both defaulting `true`, the `scrollHeight`-driven textarea, the exact `opacity-0 group-hover:opacity-100` line numbers), and cross-checking against the real files confirms all of it is accurate (except the FolderTree/Explorer conflation above).
- `PaneToggle`'s existing `aria-pressed` wiring is correctly identified as sound and the spec explicitly protects it through the mutual-exclusivity change rather than silently reworking the component.
- The State Patterns `focus-within` requirement for `Explorer.tsx:654` is the right call, and correctly recognizes `FolderTree.tsx:143` already has it — good "reuse the existing good pattern" instinct.
- Long-press-to-reveal is explicitly and correctly rejected in Inspiration & Anti-patterns as "trades one hidden affordance for another" — exactly right, and shows the spec is thinking about hidden-affordance risk generally, not just patching the one reported bug.
- Native-drag-and-drop-as-the-only-path is correctly banned with a tap-reachable fallback required ("Do"/"Don't" in DESIGN.md, Component Patterns' move-doc row) — this is the correct fix for a real touch-accessibility gap, not just a UX nicety.
- The color/contrast claim ("inherits shadcn defaults, both `:root` and `.dark` sets already exist") checks out against `knowledge/src/styles.css` — every token DESIGN.md lists has a genuine, complete dark-mode pair in the real CSS, even though DESIGN.md's own mirror only samples 4 of 8. Not a real accessibility risk.
